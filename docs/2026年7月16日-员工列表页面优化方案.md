# 员工生命周期系统：员工列表页面优化任务

## 一、任务目标

基于当前员工列表页面进行优化。

本次只调整员工列表页面及其必要的接口、数据结构和公共组件，不要重新设计整个系统，不要修改无关页面。

目标：

1. 员工列表可以直接看到员工的关键资料。
2. 减少无意义的空列和大量 `—`。
3. 将有关联的信息合并到同一列。
4. 页面不出现表格内部横向滚动条。
5. 默认页面不要出现表格内部纵向滚动条，使用分页控制数据量。
6. 修复员工姓名后面多出一个“一”或“—”的问题。
7. 展示真实的部门、岗位、入职、试用期、风险和下一事项数据。
8. 不允许通过前端逐个查询员工任职、风险或待办数据。

---

# 二、当前页面存在的问题

## 1. 关键数据缺失

当前员工列表虽然列较多，但实际大量字段为空，不能帮助 HR 快速判断员工情况。

当前缺少或展示不清晰的信息包括：

- 员工编号
- 手机号
- 部门
- 岗位
- 入职日期
- 预计转正日期
- 实际转正日期
- 试用期进度
- 生命周期状态
- 风险等级
- 下一步事项
- 下一事项截止日期
- 事项是否逾期

---

## 2. 空字段展示方式不合理

当前部门、岗位、入职信息、试用期等字段为空时，会显示多行 `—`。

问题：

- 页面看起来像数据加载失败。
- 同一列可能连续显示两个 `—`。
- 行高被无意义撑大。
- HR 无法知道是“没有数据”，还是“接口未返回”。

---

## 3. 姓名后面多出“一”或“—”

当前员工姓名后面会出现一个类似“一”的字符。

重点检查是否存在类似代码：

```vue
{{ employee.name }} {{ employee.employee_no || '—' }}
```

或：

```vue
<span>{{ employee.name }}</span>
<span>{{ employee.employee_no || '—' }}</span>
```

当员工编号为空时，占位符紧跟在姓名后面，看起来像姓名后面多了一个“一”。

必须改为：

- 姓名单独一行。
- 员工编号存在时才显示。
- 手机号存在时才显示。
- 员工编号和手机号都为空时，不渲染第二行。
- 禁止在姓名后面直接拼接空字段占位符。

---

## 4. 列过多但信息价值低

当前列表将生命周期、试用期、风险等拆成独立列，但很多列只有一个标签或一个 `—`。

这会导致：

- 横向空间浪费。
- 表格容易出现滚动条。
- 重要信息反而不突出。
- 页面视觉过于松散。

---

## 5. 操作区域占用空间过多

当前操作列长期显示：

- 查看
- 删除

问题：

- 删除属于低频、高风险操作。
- 长期显示红色删除按钮容易误点。
- 占用表格宽度。
- 后续增加编辑、异动、离职后，操作列会越来越宽。

---

# 三、推荐的最终列表结构

员工列表最终只保留以下 6 列：

| 列名 | 展示内容 |
|---|---|
| 员工 | 姓名、员工编号、手机号 |
| 任职信息 | 部门、岗位、当前任职状态 |
| 入职与试用 | 入职日期、预计/实际转正日期、试用期进度 |
| 状态与风险 | 生命周期状态、风险等级 |
| 接下来 | 下一事项、截止日期、逾期情况 |
| 操作 | 查看、更多操作 |

禁止重新增加以下独立列：

- 部门
- 岗位
- 生命周期
- 试用期
- 风险
- 入职日期
- 预计转正日期
- 创建时间

这些数据需要合并进入上述 6 列。

# 四、各列详细设计

## 1. 员工列

### 正常展示

第一行：

```text
头像  徐立伟
```

第二行：

```text
E001 · 138****1234
```

### 字段规则

- 姓名必须展示。
- 头像使用姓名首字。
- 员工编号存在时显示员工编号。
- 手机号存在时显示脱敏手机号。
- 员工编号和手机号都存在时，中间使用 `·`。
- 只有员工编号时，只显示员工编号。
- 只有手机号时，只显示手机号。
- 两者都不存在时，不显示第二行。
- 不允许在姓名旁边显示 `—`。

### 手机号脱敏规则

例如：

```text
13812341234
```

展示为：

```text
138****1234
```

手机号格式异常时：

- 不抛出错误。
- 可以原样显示，或显示 `手机号格式异常`。
- 不得影响列表加载。

---

## 2. 任职信息列

### 正常展示

第一行：

```text
直播部 · 直播运营
```

第二行：

```text
正常任职
```

### 缺失字段规则

只有部门：

```text
直播部
岗位未填写
```

只有岗位：

```text
直播运营
部门未填写
```

部门和岗位都为空：

```text
任职信息未填写
```

部门和岗位都为空时，只显示一行，不允许显示两行 `—`。

### 任职状态中文映射

必须根据后端真实枚举映射，例如：

```text
PENDING      → 待入职
ACTIVE       → 正常任职
SEPARATING   → 离职办理中
SEPARATED    → 已离职
```

禁止直接展示英文枚举。

---

## 3. 入职与试用列

根据员工当前生命周期动态展示。

### 待入职员工

```text
预计入职：2026-07-20
尚未入职
```

### 试用期员工

```text
入职：2026-04-12
转正：2026-07-12
████████░░ 82%
```

### 待转正员工

```text
入职：2026-04-12
预计转正：2026-07-12
已超期 4 天
```

### 已转正员工

```text
入职：2025-03-01
已转正：2025-06-01
```

### 离职员工

```text
入职：2024-06-01
离职：2026-07-10
```

### 日期字段规则

根据当前状态选择正确字段：

```text
待入职：
预计入职日期

已入职：
实际入职日期

试用期：
预计转正日期

已转正：
实际转正日期

已离职：
实际离职日期
```

禁止将创建员工档案的时间当作入职日期。

### 试用期进度规则

试用期进度必须基于：

```text
实际入职日期 → 预计转正日期
```

计算公式：

```text
已经过天数 ÷ 试用期总天数
```

要求：

- 最低为 0%。
- 最高为 100%。
- 已超过预计转正日期时仍显示 100%，同时显示超期天数。
- 缺少实际入职日期或预计转正日期时，不计算进度。
- 日期非法时不允许显示 `NaN%`。
- 无法计算时显示 `试用期日期未完善`。

---

## 4. 状态与风险列

该列展示两个标签：

```text
[试用期]
[低风险]
```

或：

```text
[待转正]
[中风险]
```

### 生命周期状态

统一使用现有生命周期判断方法，不允许页面自行判断。

推荐中文映射：

```text
PENDING                 → 待入职
PROBATION               → 试用期
REGULARIZATION_PENDING  → 待转正
ACTIVE                  → 正式在职
SEPARATING              → 离职办理中
SEPARATED               → 已离职
UNKNOWN                 → 状态待确认
```

### 风险等级

根据后端真实枚举映射，例如：

```text
NONE / NORMAL  → 正常
LOW            → 低风险
MEDIUM         → 中风险
HIGH           → 高风险
```

要求：

- 风险数据必须来自真实最新风险记录。
- 禁止前端写死为 `NONE` 或“正常”。
- 没有风险评估时显示 `暂未评估`。
- 正常风险使用弱化标签。
- 中高风险使用更明显的标签。
- 风险原因不要直接占满表格。
- 可以通过 Tooltip、Popover 或悬停显示风险原因。

---

## 5. 接下来列

这是员工列表中优先级最高的业务列。

### 普通待办

```text
30天跟进
7月18日 · 2天后
```

### 已逾期

```text
45天跟进
已逾期 3 天
```

### 今天到期

```text
转正面谈
今天到期
```

### 待转正

```text
转正面谈
7月20日 · 4天后
```

### 没有待处理事项

```text
暂无待处理事项
```

禁止只显示：

```text
—
```

### 下一事项选择规则

从员工当前所有未完成事项中选择最近的一条。

可参与排序的事项包括：

- 跟进任务
- 转正面谈
- 待转正确认
- 异动待生效
- 离职办理事项
- 普通待办

排序规则：

```text
1. 已逾期事项优先
2. 截止日期更早的优先
3. 截止日期相同时，按优先级排序
4. 再按事项 ID 排序，保证结果稳定
```

已完成、已取消事项不能作为下一事项。

---

## 6. 操作列

默认只显示：

```text
查看   ⋯
```

点击“查看”进入员工详情。

点击 `⋯` 打开更多操作菜单。

菜单根据员工状态动态展示：

```text
编辑员工
查看档案
记录沟通
发起异动
办理离职
删除员工
```

### 操作规则

- `删除员工` 放在菜单最下面。
- 删除项使用危险色。
- 删除必须二次确认。
- 已离职员工不显示“办理离职”。
- 待入职员工不显示不适用的试用期操作。
- 没有真实功能的菜单项必须隐藏。
- 禁止点击后只提示“功能开发中”。
- 禁止空点击事件。

# 五、表格布局与无滚动条要求

## 1. 桌面端列宽建议

```text
员工：200px
任职信息：190px
入职与试用：230px
状态与风险：150px
接下来：240px
操作：90px
```

“接下来”列允许占用剩余空间。

页面宽度足够时，表格宽度必须适配容器。

禁止给表格设置明显超过容器宽度的固定最小宽度，例如：

```css
min-width: 1400px;
```

如果当前存在此类样式，需要删除或调整。

---

## 2. 横向滚动条要求

在以下分辨率下测试：

```text
1920px
1440px
1280px
1024px
```

要求：

- 1920px、1440px、1280px 不出现表格横向滚动条。
- 1024px 下优先自动收起左侧导航。
- 搜索框和顶部操作区域可以适当缩短。
- 手机号可以只保留脱敏后四位。
- 状态标签可以纵向排列。
- 仍不允许通过增加大量独立列解决问题。

如果 1024px 实在无法容纳，可以对低优先级内容进行精简，但不要让主要内容被遮挡。

---

## 3. 纵向滚动条要求

不要给员工表格设置独立的内部纵向滚动区域，例如：

```css
max-height: 600px;
overflow-y: auto;
```

使用分页控制列表长度。

默认：

```text
每页 10 人
```

允许用户选择：

```text
10 / 20 / 50
```

默认仍为 10。

每行高度控制在：

```text
68px ～ 76px
```

列表底部显示：

```text
共 86 名员工
上一页  1 2 3 4  下一页
```

允许浏览器页面本身自然滚动，但禁止员工表格内部出现独立滚动条。

## 1. 禁止 N+1 查询

禁止前端执行：

```ts
for (const employee of employees) {
  await getEmployments(employee.id)
  await getRisk(employee.id)
  await getNextAction(employee.id)
}
```

也禁止：

```ts
await Promise.all(
  employees.map(employee => getEmployments(employee.id))
)
```

虽然并发请求，但仍属于 N+1。

员工列表需要的数据必须由后端聚合分页接口一次返回。

---

## 2. 当前任职选择规则

每名员工只能返回一条当前有效任职。

当前任职的确定规则必须读取现有后端业务规则。

不能：

- 直接取任职列表第一条。
- 直接取 ID 最大的一条但不校验状态。
- 将历史离职任职当作当前任职。
- 将未来待入职任职错误覆盖当前在职任职。

如果系统支持再次入职，需要保证当前任职判断稳定、唯一。

---

## 3. 最新风险选择规则

每名员工只返回最新一条有效风险评估。

排序建议：

```text
assessed_at DESC
created_at DESC
id DESC
```

必须过滤：

- 已删除记录
- 已失效记录
- 不属于当前员工的记录

没有风险记录时：

```json
{
  "level": null,
  "reason": null,
  "assessed_at": null
}
```

前端展示为：

```text
暂未评估
```

---

## 4. 下一事项选择规则

后端统一聚合员工未完成事项。

不能让前端分别请求：

- 跟进任务
- 待办
- 转正
- 异动
- 离职

然后自行拼装。

后端返回一条已经排序好的 `next_action`。

没有事项时返回：

```json
"next_action": null
```

# 八、空数据展示规范

统一规则：

## 单个字段为空

展示明确文字：

```text
部门未填写
岗位未填写
手机号未填写
```

## 一组字段全部为空

只显示一次：

```text
任职信息未填写
```

或：

```text
入职信息未完善
```

## 没有下一事项

显示：

```text
暂无待处理事项
```

## 没有风险评估

显示：

```text
暂未评估
```

禁止连续显示多个：

```text
—
—
```

禁止把后端 `null`、`undefined`、空字符串直接展示在页面上。

# 九、加载、空状态和错误状态

## 1. 加载状态

列表加载时使用表格骨架屏。

不要只显示：

```text
加载中...
```

筛选和翻页时：

- 保留当前表格宽度。
- 避免页面闪动。
- 禁止重复请求。
- 请求完成后恢复按钮状态。

---

## 2. 空状态

没有员工时：

```text
还没有员工档案
可以新增员工或导入花名册
```

搜索无结果时：

```text
没有找到符合条件的员工
请调整搜索词或筛选条件
```

两种空状态必须区分。

---

## 3. 错误状态

接口失败时：

```text
员工列表加载失败
[重新加载]
```

不能把接口 500、网络错误或超时展示成“暂无员工”。

# 十一、日期展示规则

统一使用系统日期工具，不允许各组件自行拼接日期。

推荐格式：

```text
完整日期：2026-07-18
列表短日期：7月18日
```

相对日期：

```text
今天到期
明天到期
2天后
已逾期3天
```

计算日期时按当天零点处理。

禁止使用 `Math.round` 计算天数。

需要正确处理：

- 包含时分秒的时间
- 纯 `YYYY-MM-DD`
- 跨月
- 跨年
- 时区
- 非法日期
- 空日期

# 十二、测试要求

# 十三、验收标准

必须同时满足：

1. 员工姓名后面不再出现“一”或 `—`。
2. 员工编号和手机号只在存在时显示。
3. 员工列表最终只保留 6 个主列。
4. 部门和岗位合并在“任职信息”列。
5. 入职日期、转正日期和试用期进度合并展示。
6. 生命周期和风险合并在一列。
7. 下一事项展示真实数据。
8. 没有下一事项时显示明确文案。
9. 大量连续 `—` 被清理。
10. 删除操作进入更多菜单。
11. 删除员工需要二次确认。
12. 1280px 及以上无横向滚动条。
13. 表格内部无纵向滚动条。
14. 默认每页 10 人。
15. 列表不产生 N+1 请求。
16. 风险数据不是前端写死。
17. 部门和岗位来自当前有效任职。
18. 入职日期使用实际入职日期。
19. 页面刷新后筛选条件保留。
20. 搜索支持防抖。
21. 接口错误不会显示成空数据。
22. `npm run build` 成功。
23. 现有员工详情、导入、试用期、转正等功能不受影响。

# 十四、禁止事项

- 不要重新设计整个系统。
- 不要修改无关页面。
- 不要新增大量独立列。
- 不要通过横向滚动条容纳字段。
- 不要给表格增加内部纵向滚动条。
- 不要在姓名后拼接空字段占位符。
- 不要连续显示多个 `—`。
- 不要将创建时间当作入职时间。
- 不要将风险等级写死为正常。
- 不要由前端逐个查询员工任职。
- 不要由前端逐个查询风险。
- 不要由前端逐个查询下一事项。
- 不要保留空点击按钮。
- 不要直接展示英文枚举。
- 不要为了页面好看而伪造数据。
- 不要破坏花名册导入和员工详情功能。
- 不要只修改样式，不验证真实接口数据。
- 不要只回复“已完成”。

# 十五、执行要求

完成后执行：

## 前端

```bash
cd frontend
npm install
npm run test
npm run build
```

如果项目尚未配置测试框架，至少执行：

```bash
npm run build
```

并手动完成页面验收。

## 后端

执行当前项目已有的测试命令。

如果修改了数据库或增加索引：

- 必须提供迁移。
- 必须验证迁移升级。
- 必须验证迁移回滚。
- 不允许直接修改生产数据库文件。
