# 员工生命周期系统前端重设计与实施方案

> 仓库：`https://github.com/stonelight1/employee-lifecycle.git`  
> 审查分支：`master`  
> 审查日期：2026-07-16  
> 文档用途：可直接交给 Codex 执行前端重构  
> 适用场景：单公司、本地单机部署、员工规模 100 人以内、主要使用者为 1 名 HR

---

## 0. 执行结论

当前系统的主要问题不是单纯“颜色不好看”，而是仍停留在传统 CRUD 后台的页面组织方式：

- 进入系统后直接看到员工表格，而不是 HR 今天需要处理的事情。
- 试用期、跟进、转正、异动、离职等核心生命周期能力被拆散在多个深层页面中。
- 页面普遍采用“白色卡片 + 表格 + 蓝色按钮”，不同业务的重要程度没有视觉差异。
- 员工列表更像通讯录，无法快速判断员工处于哪个阶段、是否有风险、下一步要做什么。
- 跟进任务和待办事项是两张相似的表格，HR 需要自己判断优先级。
- 员工详情只有基本信息、任职记录和未完成的时间线，没有形成员工完整生命周期档案。
- 页面缺少统一组件、统一状态颜色、统一交互反馈和统一设计变量。
- 原生 `alert`、`confirm`、`prompt` 较多，整体体验偏开发工具，不像完成度较高的业务系统。
- 部分页面直接显示后端英文状态值，业务理解成本较高。
- 当前前端没有组件库、图标库和图表库，几乎所有样式都在各页面中重复编写。

本次重设计的核心不是“做得更花”，而是把系统从：

> 员工数据管理后台

升级为：

> HR 每日工作台 + 员工生命周期驾驶舱

# 1. 当前前端审查结果

## 1.1 当前技术结构

前端技术栈：

- Vue 3
- TypeScript
- Vite
- Vue Router
- Pinia
- Axios
- Day.js

当前没有引入：

- Element Plus 等完整 UI 组件库
- 图标库
- 图表库
- 全局设计变量体系
- 通用表格、弹窗、状态标签、空状态等业务组件

本次建议继续保持轻量，不引入大型 UI 框架，只新增：

```json
{
  "lucide-vue-next": "^0.468.0"
}
```

图标统一使用 `lucide-vue-next`，禁止继续混用空字符串、Emoji 和文本箭头。

---

## 1.2 页面结构问题

### 1.2.1 缺少工作台

当前路由 `/` 直接跳转 `/employees`。

这意味着用户每次进入系统，第一眼看到的是员工列表，而不是：

- 今天有几项待办
- 哪些事项已经逾期
- 哪些员工即将转正
- 哪些员工存在风险
- 最近 7 天有哪些关键节点
- 当前公司员工处于哪些生命周期阶段

这是当前系统没有侧重点的根本原因。

### 1.2.2 左侧导航按技术模块组织，而不是按 HR 工作组织

当前一级导航：

- 员工管理
- 跟进任务
- 节点配置
- 待办事项
- 操作日志

存在以下问题：

- “节点配置”和“操作日志”属于低频管理功能，却和核心业务并列。
- 试用期、转正、异动、离职没有一级入口。
- 跟进任务与待办事项高度相似，用户需要来回切换。
- 没有“今日工作”“即将到期”“风险员工”等业务入口。
- 页面导航没有体现员工从入职到离职的生命周期主线。

### 1.2.3 员工列表信息价值过低

当前员工列表主要展示：

- 姓名
- 员工编号
- 手机号
- 状态
- 创建时间
- 操作

这些字段只能用于查找员工，无法支持 HR 做判断。

员工列表应该优先展示：

- 部门、岗位
- 入职日期
- 当前生命周期阶段
- 试用期进度
- 预计转正日期
- 当前风险等级
- 最近一次跟进
- 下一项待处理事项

“创建时间”不是 HR 日常最关心的信息，不应占据主列表核心列。

### 1.2.4 员工详情缺少“员工 360° 档案”

当前员工详情页采用三个标签：

- 基本信息
- 任职记录
- 时间线

主要问题：

- 页面顶部只显示姓名和状态，缺少部门、岗位、入职时间、试用期进度、风险等核心上下文。
- 任职、沟通、跟进、试用期评估、转正、异动、离职分散在不同路由。
- 时间线页面仍为“功能开发中”。
- 页面没有明确告诉 HR 当前员工下一步应该做什么。
- 缺少从当前页面快速执行“记录沟通、完成跟进、发起转正、发起异动、启动离职”等操作的入口。
- 当前代码加载任职列表后没有将返回数据赋值给 `employments`，会导致任职记录持续显示为空，应在重构时一并修复。

### 1.2.5 跟进任务和待办事项缺少优先级表达

当前两个页面均为普通表格。

跟进任务列表没有直接展示：

- 对应员工
- 部门、岗位
- 是否逾期
- 距计划日期还有几天
- 最近沟通时间
- 风险等级
- 推荐动作

待办事项列表没有按：

- 已逾期
- 今天
- 未来 7 天
- 更晚

进行分组。

用户必须逐行阅读日期，无法快速识别工作重点。

### 1.2.6 生命周期流程页面没有流程感

试用期、转正、异动、离职页面目前大多采用：

> 页面标题 → 多个白色卡片 → 表单 → 按钮 → 历史表格

虽然能完成操作，但没有体现业务阶段，也没有清晰表达：

- 当前位于哪一步
- 上一步是什么
- 下一步是什么
- 当前是否存在阻塞
- 操作会影响哪些数据
- 完成后员工状态如何变化

### 1.2.7 样式重复且缺少统一规则

多个页面重复定义：

- `.table-card`
- `.card`
- `.btn`
- `.btn-primary`
- `.btn-danger`
- `.tag`
- `.page-header`
- `.info-grid`
- `.empty`

重复带来的问题：

- 修改主题时需要逐页改动。
- 同一种组件在不同页面出现细微差异。
- 按钮、标签、间距和阴影没有统一标准。
- 页面越多，视觉差异越明显。

### 1.2.8 原生浏览器交互影响完成度

当前存在较多：

```ts
alert(...)
confirm(...)
prompt(...)
location.reload()
```

需要统一替换为：

- Toast 消息
- 确认弹窗
- 表单弹窗
- 局部数据刷新
- 按钮 Loading 状态
- 错误提示区域

### 1.2.9 状态展示没有业务翻译

例如页面直接显示：

- `PENDING`
- `IN_PROGRESS`
- `ACTIVE`
- `SEPARATING`
- `ALL`
- `SINGLE`

要求统一通过状态映射函数转为中文，禁止模板中直接展示后端枚举。

# 2. 新版设计定位

## 2.1 产品定位

新版前端定位为：

> 面向单人 HR 的员工生命周期工作台

不是大型集团 HR SaaS，不追求复杂菜单和大屏数据，而是强调：

1. 今天要做什么
2. 哪些员工需要关注
3. 员工目前处于哪个阶段
4. 下一步该执行什么动作
5. 历史过程是否清晰可追溯

---

## 2.2 设计主题

设计主题名称：

# Lifecycle Flow · 生命周期流

视觉核心是“阶段流转”，而不是“表格管理”。

关键视觉元素：

- 生命周期步骤条
- 试用期进度条
- 工作事项时间分组
- 风险状态点
- 员工时间线
- 前后信息对比
- 下一步动作卡片

---

## 2.3 设计原则

### 原则一：行动优先

首页优先展示需要处理的事项，而不是静态统计。

### 原则二：员工上下文不丢失

进入试用期、转正、异动、离职等页面时，顶部始终显示员工姓名、部门、岗位、入职日期和当前状态。

### 原则三：状态可感知

状态不能只通过文字表达，必须同时通过：

- 颜色
- 标签
- 图标
- 位置
- 进度

表达。

### 原则四：危险操作弱暴露

删除、确认离职等危险操作不作为页面主按钮，应放入更多菜单或独立危险区域。

### 原则五：单人使用，减少无意义功能

系统只有一名 HR 使用，因此不需要：

- 用户头像下拉菜单
- 权限入口
- 角色切换
- 消息中心
- 在线状态
- 多租户切换

顶部“本地 HR”文字没有实际价值，应删除。

### 原则六：轻量但不简陋

不引入复杂大型 UI 框架，但必须建立自己的基础组件体系。

# 6. 核心创新设计

## 6.1 生命周期轨道

每名员工详情顶部显示统一生命周期轨道：

```text
待入职 ── 试用期 ── 待转正 ── 正式在职 ── 异动 ── 离职
              ● 当前
```

规则：

- 已完成阶段：实心主色
- 当前阶段：主色高亮 + 外环
- 未到达阶段：灰色
- 非线性阶段，如异动，可以显示为正式在职阶段下的事件节点
- 已离职员工的历史阶段仍然保留

该组件命名：

```text
LifecycleStepper.vue
```

---

## 6.2 下一步行动卡

员工详情页必须有“下一步行动”区域。

示例：

```text
下一步行动
────────────────────────────
7 天跟进已逾期 2 天
建议尽快完成沟通并记录结果

[立即处理] [稍后提醒]
```

优先级：

1. 已逾期事项
2. 今天到期事项
3. 未来 7 天事项
4. 待转正
5. 待确认异动或离职
6. 无待办时显示“当前暂无需处理事项”

---

## 6.3 工作事项时间分组

工作事项不再仅使用普通表格。

分组顺序：

```text
已逾期 3
今天 5
未来 7 天 8
更晚 12
```

每条事项展示：

- 员工头像或姓名首字
- 员工姓名
- 部门 / 岗位
- 事项名称
- 到期时间
- 事项来源
- 风险等级
- 快速完成按钮

---

## 6.4 员工快速预览抽屉

员工列表点击姓名后，先打开右侧预览抽屉。

抽屉内容：

- 姓名、部门、岗位
- 当前状态
- 入职日期
- 试用期进度
- 风险等级
- 下一项任务
- 最近沟通摘要
- “打开完整档案”按钮

这样用户可以快速查看，不必频繁跳转页面。

完整详情仍保留独立路由。

---

## 6.5 前后信息对比

员工异动页面不能仅显示输入表单。

需要显示：

```text
异动前                          异动后
────────────────────────────────────────
电商部                          直播部
招商主管                        运营主管
武汉                            武汉
2026-06-01 生效                 2026-07-20 生效
```

发生变化的字段高亮，未变化字段弱化。

组件命名：

```text
ChangeDiffPanel.vue
```

# 22. 最终目标

重构完成后，系统应具备以下感受：

- 打开系统后立即知道今天要做什么。
- 不需要进入多个页面查找即将转正、逾期和风险员工。
- 员工列表能直接判断员工当前阶段和下一步动作。
- 员工详情能看到完整生命周期，而不是零散数据。
- 转正、异动和离职具备清晰流程感。
- 全系统状态、按钮、表单、弹窗和空状态一致。
- 页面具有明显业务侧重点，而不是所有信息同等展示。
- 视觉风格现代、克制、清晰，不追求花哨动画。
- 保持本地单机系统易维护、低依赖和低复杂度。
