# 04 单用户访问与安全规则

> 版本：v0.4.0  
> 最后更新：2026-07-16  
> 文件名为兼容既有文档引用继续保留 `04-permission-rules.md`，V1 不再实现权限认证。

## 1. 已确认结论

本系统只由一名 HR 使用。V1 删除应用内权限认证功能，不实现：

- 登录、退出和找回密码。
- 用户注册和用户管理。
- 密码、验证码和多因素认证。
- JWT、Access Token、Refresh Token 和 Session。
- `HR_ADMIN`、`HR_MEMBER`、`HR_VIEWER` 等角色。
- 权限点、角色权限映射和个人权限覆盖。
- 部门、员工或任务数据范围。
- 401、403 和前端权限按钮控制。

删除权限认证不改变员工、任务、沟通、AI、风险、转正、离职、日志等业务规则。

## 2. 单用户操作者

后端通过环境配置维护固定操作者：

- `LOCAL_OPERATOR_ID`：默认 `1`。
- `LOCAL_OPERATOR_NAME`：默认 `LOCAL_HR`，部署时可以改为实际显示名称。

规则：

1. 所有创建、修改、删除、确认、审核和日志动作均由后端写入固定操作者。
2. 前端不得传入或覆盖操作者 ID、姓名、角色和权限。
3. 请求体中出现 `created_by`、`updated_by`、`operator_id`、`confirmed_by`、`reviewed_by` 等字段时，后端应忽略或返回参数错误。
4. 操作日志继续保留，操作者用于本地审计，不表示经过登录认证。
5. 现有业务字段可以继续保存固定 ID，以减少数据库和代码改动。

## 3. 前端要求

必须删除或停用：

- 登录页、退出按钮和个人中心中的账号安全入口。
- 用户、角色和权限管理菜单。
- 登录状态 Store。
- Token 保存、刷新和过期处理。
- Axios 请求头自动附加 Token 的逻辑。
- 401/403 拦截、重定向登录和无权限页面。
- 路由 `meta.permission`、权限指令和权限按钮组件。

页面启动后直接进入系统首页。业务按钮是否可用只根据数据状态和业务规则判断。

## 4. 后端要求

必须删除或停用：

- 登录、退出、刷新令牌和当前用户接口。
- 用户、角色、权限点及映射接口。
- JWT、Session、密码哈希和验证码依赖。
- `get_current_user`、`require_permission`、`require_role`、数据范围过滤等依赖。
- 认证中间件和基于用户身份的全局拦截器。
- 仅为鉴权服务的缓存、表和定时清理任务。

必须保留：

- 参数校验。
- 资源存在性和逻辑删除校验。
- 状态流转校验。
- 幂等和版本冲突控制。
- 业务对象关系校验。
- 操作日志。
- AI 输入版本和返回 Schema 校验。

不得通过编造一个“永远拥有所有权限的管理员”来保留整套鉴权链路。应真正移除认证依赖，避免后续维护无效代码。

## 5. 数据库处理

### 5.1 未实现认证表

若认证功能尚未开发，不创建任何用户、角色、权限、会话和令牌表。

### 5.2 已存在认证表

若代码或数据库中已经存在相关结构：

1. 先搜索业务表是否存在外键或查询依赖。
2. 停止新写入和接口使用。
3. 对无历史价值且未上线的数据，可通过迁移删除。
4. 对已有历史数据，优先只读保留，待确认无依赖后再删除。
5. 所有删除必须提供正向迁移、回滚和备份说明。

### 5.3 审计字段

`created_by`、`updated_by`、`deleted_by`、`user_id`、`hr_user_id`、`confirmed_by`、`reviewed_by` 和 `operator_id` 可以继续存在，但：

- 不要求关联用户账号表。
- 后端统一写固定本地操作者。
- 不参与数据过滤和权限判断。
- 旧字段类型优先保持不变，避免无必要重构。

## 6. 网络安全边界

应用内没有认证，必须通过部署边界降低风险：

- 本机使用时后端默认监听 `127.0.0.1`。
- 内网使用时仅监听受信任网卡，并由防火墙限制来源。
- 不得将后端端口、SQLite 文件、备份目录或日志目录直接暴露到公网。
- CORS 只允许实际前端地址，不允许生产环境使用任意来源。
- 远程访问优先使用 VPN、公司安全网关或带认证的反向代理。
- 未增加外部保护前，不允许通过公网域名直接访问系统。

外部网关认证属于部署层保护，不改变本系统单用户、无账号的产品设计。

## 7. AI 接口访问

AI 接口不做用户权限校验，但仍必须：

- 只接受系统内存在的 `communication_id` 和当前文本版本 ID。
- 不接受前端直接提交任意员工文本绕过沟通记录。
- 校验沟通记录、任职和文本版本之间的归属关系。
- 校验幂等键、处理中状态、当前版本和文本长度。
- 从服务端环境变量读取模型密钥。
- 不在响应或日志中泄露密钥和完整敏感文本。

## 8. 验收标准

- 启动系统后直接进入首页。
- 不存在登录、退出、用户、角色和权限页面。
- 前端不保存或发送 Token。
- 后端业务接口无认证请求头也可正常调用。
- 项目不存在实际生效的认证中间件和权限依赖。
- API 不返回 401 或 403。
- 操作者字段由后端固定写入，前端无法伪造。
- 操作日志仍能记录固定本地操作者和业务变化。
- 业务校验、幂等、并发和状态机仍正常。
- 默认部署地址不直接对公网开放。

## 9. 未来扩展

未来确实出现多人使用、远程协作或数据隔离需求时，应作为独立版本重新设计：

- 用户与登录方式。
- 角色和权限点。
- 数据范围。
- 密码和令牌安全。
- 审计主体迁移。
- 现有固定操作者字段的数据迁移。

不得在 V1 中保留半套不可验证的鉴权代码。
